# 如何确认已经使用 PGVector 检索增强

这篇文档只讲一件事：怎么判断应用的回答，真的用到了 PGVector 里的知识，而不是模型自己“猜”出来的。

## 一个好理解的类比

可以把整个 RAG 流程想成一次开卷考试：

- 大模型：像答题的学生。
- PGVector：像考场里提前准备好的资料柜。
- 用户问题：像试卷上的题目。

如果学生没有去翻资料柜，只靠自己发挥，那就不算用了知识库。
只有先从资料柜里找到相关资料，再把资料带进答题过程，最后答案里体现出这些资料，才算真正用了 RAG。

## 只看最终回答，为什么还不够

如果只看到最后返回了一段回答，我们最多只能证明：

- 应用没有报错。
- 模型给出了内容。

但这还不能证明模型真的查了 PGVector。
因为模型有时会自己生成一段看起来很合理的话，就像学生不翻书也能写几句答案一样。

## 更可靠的验证思路

要想确认真的用了 PGVector，最好分成两步验证。

### 第一步：先证明 PGVector 自己能查到数据

这一步不走最终聊天回答，只直接调用：

```java
pgVectorVectorStore.similaritySearch(...)
```

如果这里能查回预期文档，说明：

- 文档已经成功写入 PostgreSQL。
- 向量已经成功生成。
- PGVector 检索本身是通的。

这一步相当于先确认资料柜里确实有资料，而且能找得到。

### 第二步：再证明聊天回答真的用了这些检索结果

当前项目里的做法是：

1. 先用 `pgVectorVectorStore.similaritySearch(...)` 按用户问题取回前 3 条文档。
2. 把这些文档文本拼成一段知识库上下文。
3. 再把上下文一起放进最终的 system prompt。
4. 最后让模型基于用户问题和检索上下文生成回答。

也就是说，流程从先查知识库变成了先查，再回答。

## 这次排查时，我们是怎么验证的

为了避免中文语义和回答风格干扰，我们临时放入了一个只存在于知识库里的唯一标识，例如：

```text
RAG_PROOF_TOKEN_MARRIAGE_ALPHA_2026
```

然后做两层验证：

1. 直接查 PGVector，看能不能把这个唯一标识查回来。
2. 再问模型一个不直接包含标识的问题，例如：

```text
What is the verification token mentioned in the knowledge base?
Reply with the exact token only.
```

如果最终回答还能准确返回这个 token，就说明模型不是在复读用户输入，而是真的拿到了知识库里的内容。

## 这次验证成功时，说明了什么

当最终回答精确返回知识库中的 token 时，可以认为下面这条链路已经打通：

```text
用户问题
-> PGVector 相似度检索
-> 取回知识库文档
-> 拼接到提示词上下文
-> 大模型生成最终回答
```

这就说明应用已经不是只有模型在说话，而是模型先查知识库，再回答。

## 为什么会看到重复文档

之前项目里出现过明明只有 3 个 Markdown 文件，结果却查到重复文档的现象。

原因不是文件真的重复了，而是有两个点容易让人误解：

### 1. 一个 Markdown 文件，不一定只生成一条文档

`MarkdownDocumentReader` 会按标题和内容结构，把一个 Markdown 文件拆成多条 `Document`。
所以 3 个 Markdown 文件，最后完全可能变成十几条文档。

### 2. 应用每次启动时都可能再次导入知识库

如果在创建 `pgVectorVectorStore` 时直接执行：

```java
vectorStore.add(documents);
```

那么每次 Spring Boot 启动，都会重新往表里插入一遍。
由于每条记录都有新的 id，相同内容不会覆盖旧数据，而是会变成新的重复记录。

## 现在项目里怎么避免重复导入

当前项目已经改成：

1. 先查询 `public.vector_store` 当前行数。
2. 如果表里已经有数据，就跳过初始化导入。
3. 只有表为空时，才导入知识库文档。

这样可以避免每次启动都重复写入同一批内容。

## 实际排查时，建议按什么顺序看

如果你以后怀疑是不是没用上 PGVector，推荐按下面顺序检查：

1. 先看 `pgVectorVectorStore.similaritySearch(...)` 是否能查回预期文档。
2. 再看日志里的 `RAG retrieved documents count` 和 `RAG retrieved document preview`。
3. 最后看最终回答里，是否真的出现了只存在于知识库里的关键内容。

## 一句话总结

判断有没有真正用上 PGVector，不是看回答像不像，而是看：

**能不能先从向量库检索出文档，再让最终回答体现出这些文档中的关键信息。**
